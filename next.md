1. the environment should be set up from its glb file, and random rocks are to be generated in the environment using a script. The script should allow for customization of the number and size of rocks, as well as their placement within the environment(number of rocks, size range, placement constraints, how far from each other must they stay). The script should make sure to generate rocks that do not intersect with each other or with the environment boundaries, however are at the terrain y level. Additionally, keep the position and orientation of rocks, since during ablation we need to keep the same environment for consistency.
2. so a simpilified environment generation flow is: 
   - Load the environment from its glb filse.
   - Generate random rocks based on user-defined parameters (number, size, placement).
   - Ensure that the position and orientation of rocks remain consistent across different runs for ablation studies.
   - Save the generated environment for use in experiments.
   - Load the generated environment later for experiments rather than regenerating it each time, to maintain consistency in the experimental setup.
3. keep both 'goal from vlm' and 'goal uv', but default to goal from vlm, since that is the study. 
4. the scene should be loaded with the generated rocks and the environment should be ready for experiments. The rocks should be placed in such a way that they do not interfere with the main objectives of the experiment, while still providing a realistic and challenging environment for testing.
5. the scene should be loaded in, the agent is to be created with RGB, Depth and semantic cameras. the agent's position should follow the y level of the terrain and whatever position the agent is in. let's say the agent is spawned in at y +2 meters above the terrain level, it should always maintian +2 meters above the terrain level, regardless of the terrain's height variations. for this specific task, agent should have a function to derive terrain height at its current x, z position and adjust its y position accordingly to maintain the +2 meters offset, from the terrain heightmap image. 
6. driving policy and navigation should be copied over from @rollout_navdp_policy.py. make sure to copy over it's cbf and other necessary functions to ensure the agent can navigate the environment effectively. The navigation policy should be able to handle the generated rocks and any other obstacles present in the environment, while still achieving the main objectives of the experiment.
7. the scene should be set up with the agent and its sensors, then the first frame is to be sent to the custom SAM model to be segmented. After which the segmentation results should be used to identify exactly ONE goal object in the scene. all other rocks are to be defined as obstacles and should be avoided by the agent during navigation using the navigation policy's CBF rather than building avoidance from scratch. 
8. qwen is to be loaded in the background as a server and let standby for requests. it should expect only two types of requests:
    - goal selection request: inputs (image, segmentation results bboxes), outputs (selected goal object id, reasoning for selection in json)
    - action suggestion: inputs(current frame, belief goal position), outputs(a single word action: left, right, forward, stop)
9. the NavDP policy should remain the primary navigation controller, generating the nominal trajectory toward the selected goal while relying on its existing Control Barrier Function (CBF) for safe obstacle avoidance. Qwen should be queried only under two conditions: (1) when one or more obstacles lie within a configurable obstacle distance threshold X of the rover, or (2) when the rover is within a configurable goal distance threshold Y of the selected goal. Each query should provide Qwen with the current RGB frame together with the estimated relative goal position (distance and yaw). Based on this information, Qwen should return a single high level steering action: left, right, forward, or stop. The returned action serves only as a directional preference for the NavDP policy and does not replace the underlying CBF-based controller. For example, if the goal lies approximately straight ahead (0° yaw), either left or right may be reasonable for bypassing a nearby obstacle, whereas if the goal lies significantly to one side (for example, +120° yaw), Qwen should prefer the direction that results in a shorter path toward the goal. When the rover enters the goal proximity threshold Y, Qwen should continue issuing steering actions until it determines that the rover is correctly aligned with the goal, at which point it should return stop. Receipt of the stop action marks successful completion of the episode.
10. each frame should have the goal and obstacles highlighted, since we already have their meshes, which we get by segment using SAM -> select bbox to actual world coords -> select a zone about 0.5m radius around the coords -> extract the mesh of the goal and obstacles. this will help visualize the agent's perception of the environment and its decision-making process during navigation. The highlighted meshes should be rendered in a way that clearly distinguishes between the goal and obstacles, allowing for easy identification during analysis of the experiment results.
11. the driving behaviour can be changed based on the flags used during launch. 
12. log all relevant data during the experiment, including per frame: agent position, orientation, distance to goal, action; per episode config: goal mode, steering mode, obstacle seed/count, CBF on/off; per episode summary: success, .
